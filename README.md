# AgenticQA Workflow

A template and reference guide for QA engineers to run a complete, AI-assisted end-to-end testing workflow — from user story to committed test artifacts — using Claude Code, Playwright, and MCP servers.

---

## What Is This?

This project is a **reusable QA workflow template** that combines AI agents with browser automation to cover the full testing lifecycle:

1. Read a user story
2. Explore the application and create a test plan
3. Perform manual exploratory testing
4. Generate automated Playwright test scripts
5. Execute tests and auto-heal failures
6. Produce a test execution report
7. Commit all artifacts to your repository

The workflow is driven by prompts in [End2EndPromptFile.md](End2EndPromptFile.md) — copy those prompts into Claude Code one step at a time and the agents do the heavy lifting.

## Setup

### 1. Clone or fork this repository

```bash
git clone https://github.com/NBosch-code/AgenticQA_workflow.git
cd AgenticQA_workflow
```

Or use it as a template by creating a new repository from it on GitHub and cloning your copy.

### 2. Install dependencies

```bash
npm ci
```

### 3. Install Playwright browsers

```bash
npx playwright install --with-deps
```

This installs Chromium, Firefox, and WebKit along with their system dependencies.

### 4. Open in VS Code with Claude Code

Open the project folder in VS Code. Claude Code picks up the MCP server configuration from [.vscode/mcp.json](.vscode/mcp.json) automatically, which registers:

- **playwright-test** — MCP server used by the AI agents to interact with the browser
- **playwright** — General-purpose Playwright MCP for exploratory steps
- **github** — GitHub Copilot integration

> Make sure the Claude Code VS Code extension is installed and you are signed in.

## Adapting the Template for Your Own User Story

### Step 1: Replace the user story

Create a new markdown file in `.vscode/user-stories/` describing your feature under test. Follow the structure of the example:

```
.vscode/user-stories/your-feature.md
```

Include:
- Application URL
- Test credentials (if applicable)
- Acceptance criteria (AC1, AC2, etc.)
- Scope and assumptions

### Step 2: Update the End2EndPromptFile prompts

Open [End2EndPromptFile.md](End2EndPromptFile.md) and update any references to the example user story filename (`jira-task-checkout.md`) to point to your new file.

### Step 3: Follow the 7-step workflow below

## Running the Workflow

The entire workflow is driven by prompts in [End2EndPromptFile.md](End2EndPromptFile.md). Open Claude Code (in VS Code or the terminal) and paste each prompt in sequence.

### Step 1 — Read the User Story

Paste this prompt into Claude Code:

```
I need to start a new testing workflow. Please read the user story from the file:
.vscode/user-stories/jira-task-checkout.md

Summarize the key requirements, acceptance criteria, and testing scope.
```

**Output:** A summary of requirements, acceptance criteria, URL, and credentials.

---

### Step 2 — Create a Test Plan

```
Based on the JIRA user story jira-task-checkout.md that we just reviewed, use the playwright-test-planner agent to:
1. Read the application URL and test credentials from the user story
2. Explore the application and understand all workflows mentioned in the acceptance criteria
3. Create a comprehensive test plan that covers all acceptance criteria including:
    - Happy path scenarios
    - Negative scenarios (validation errors, empty fields, invalid data)
    - Edge cases and boundary conditions
    - Navigation flow tests
    - UI element validation
4. Save the test plan as a markdown file in the specs folder: specs/saucedemo-test-plan.md

Ensure each test scenario includes:
    - Clear test case title
    - Detailed step-by-step instructions
    - Test data requirements
```

**Output:** A test plan saved to `specs/` with organised scenarios.

**Agent used:** `playwright-test-planner` — navigates the live application, maps user flows, and designs test scenarios.

---

### Step 3 — Perform Exploratory Testing

```
Now I need to perform manual exploratory testing using Playwright CLI.
Please read the test plan from: specs/saucedemo-test-plan.md

Then execute the test scenarios defined in the test plan:
1. Use Playwright browser tools to manually execute each test scenario from the plan
2. Follow the step-by-step instructions in each test case
3. Verify expected results match actual results
4. Take screenshots at key steps and error states
5. Document your findings:
    - Test execution results for each scenario
    - Any UI inconsistencies or unexpected behaviours
    - Missing validations or bugs discovered
    - Screenshots as evidence
```

**Output:** Manual test results, screenshots, and a list of observations.

---

### Step 4 — Generate Automation Scripts

```
Now I need to create automated test scripts using the playwright-test-generator agent.

Please review:
1. Test Plan from: specs/saucedemo-test-plan.md
2. Exploratory testing results from Step 3 (for actual element selectors and UI insights)

Using insights from the manual exploratory testing:
1. Leverage the element selectors and locators that were successfully used in Step 3
2. Use stable element properties (IDs, data attributes, roles) discovered during exploration
3. Apply wait strategies and UI behaviours observed during manual testing
4. Incorporate any workarounds for UI quirks discovered

Generate Playwright JavaScript automation scripts:
1. Create tests for each test scenario from the test plan
2. Organise scripts into appropriate test suite files in tests/
3. Use the test case names from the test plan and align with the way this project is set up
4. Use reliable selectors and strategies from exploratory testing

Requirements for all scripts:
1. Follow Playwright best practices
2. Include proper assertions using expect()
3. Use descriptive test names matching the format in the test plan
4. Use robust element selectors discovered during manual testing
5. Add comments for complex steps
6. Use proper wait strategies based on actual application behaviour
7. Add proper test hooks (beforeEach, afterEach)

After generating the scripts, run the tests to verify they pass.
```

**Output:** Test files created in `tests/`.

**Agent used:** `playwright-test-generator` — generates Playwright scripts from the test plan and selector knowledge gathered during exploration.

---

### Step 5 — Execute and Heal Tests

```
Now I need to execute the generated automation scripts and heal any failures using the playwright-test-healer agent.

1. Run all automation scripts for this user story in tests/
2. Identify any failing tests
3. For each failing test, use the playwright-test-healer agent to:
   - Analyse the failure (selector issue, timing issue, assertion failure)
   - Auto-heal the test by fixing selectors, adding waits, or adjusting assertions
   - Update the test scripts with the fixes
4. Re-run the healed tests to verify they pass
5. Repeat the heal process until all tests are stable and passing
6. Document:
   - Initial test results (pass/fail count)
   - Healing activities performed
   - Final test results after healing
   - Any tests that could not be auto-healed
```

**Output:** All tests passing, with healed scripts in `tests/`.

**Agent used:** `playwright-test-healer` — diagnoses test failures and applies targeted fixes.

---

### Step 6 — Create a Test Report

```
Now I need to create a comprehensive test execution report based on manual testing, automation execution, and healing activities.

Please compile results from:
- Step 3: Manual exploratory testing results
- Step 4: Generated automation scripts
- Step 5: Automated test execution and healing results

Structure the report as a markdown file in the test-results folder.

Include:
1. Executive Summary
    - Total test cases planned
    - Test cases executed (manual + automated)
    - Overall Pass/Fail/Blocked status

2. Manual Testing Results
    - Results from Step 3 exploratory testing
    - Screenshots and observations
    - Issues found during manual testing

3. Automated Test Results
    - Initial automation results from Step 5
    - Healing activities performed
    - Final test execution results after healing
    - Test suite execution summary

4. Defects Log
    - Bug ID, Severity, Title, Description
    - Steps to Reproduce
    - Expected vs Actual Behaviour
    - Screenshots/Evidence

5. Test Coverage Analysis
    - Which acceptance criteria are covered
    - Coverage from manual vs automated tests
    - Any gaps and recommendations

6. Summary and Recommendations
```

**Output:** A full test report saved to `test-results/`.

---

### Step 7 — Commit to Repository

```
Now I need to commit all the test artifacts to the repository.

Repository URL: <your-repo-url>

Please perform the following Git operations:

1. Initialize Git repository if not already initialized
2. Stage all files in the workspace (all new and modified files)
3. Create a commit with the message:
   "feat(tests): Add complete test suite for <user story name>

    - Add user story documentation
    - Add comprehensive test plan with all scenarios
    - Add test execution report with results
    - Add automated test scripts
    - Include validation, navigation, and edge case tests

    Resolves <ticket-id>"

4. Push all changes to the Git repository
```

**Output:** All artifacts committed and pushed to your repository.

## Running Tests Manually

You can run the Playwright tests at any time without the AI workflow:

```bash
# Run all tests across all browsers
npx playwright test

# Run tests in a specific browser only
npx playwright test --project=chromium

# Run a specific test file
npx playwright test tests/checkout.spec.ts

# Run tests in headed mode (see the browser)
npx playwright test --headed

# Open the HTML report after a run
npx playwright show-report
```

Test results are written to `playwright-report/` (HTML) and `test-results/` (JSON/traces).

## AI Agents Reference

Three agents are defined in [.github/agents/](.github/agents/). They are invoked automatically when you use their name in a prompt inside Claude Code.

| Agent | File | Purpose |
|-------|------|---------|
| `playwright-test-planner` | [playwright-test-planner.agent.md](.github/agents/playwright-test-planner.agent.md) | Explores the live application and creates a comprehensive test plan |
| `playwright-test-generator` | [playwright-test-generator.agent.md](.github/agents/playwright-test-generator.agent.md) | Generates Playwright scripts from the test plan and discovered selectors |
| `playwright-test-healer` | [playwright-test-healer.agent.md](.github/agents/playwright-test-healer.agent.md) | Diagnoses failing tests and applies fixes automatically |

All agents use the `playwright-test` MCP server (`npx playwright run-test-mcp-server`) to control a real browser during their work.

## MCP Server Configuration

MCP servers are configured in [.vscode/mcp.json](.vscode/mcp.json). VS Code and Claude Code load this automatically when the project is open.

```json
{
  "servers": {
    "playwright-test": {
      "type": "stdio",
      "command": "npx",
      "args": ["playwright", "run-test-mcp-server"]
    },
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"],
      "type": "stdio"
    },
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/"
    }
  }
}
```

No additional installation is needed — `npx` pulls these on first use.

## CI/CD

The GitHub Actions workflow at [.github/workflows/playwright.yml](.github/workflows/playwright.yml) runs automatically on every push or pull request to `main`/`master`:

1. Checkout code
2. Set up Node.js (LTS)
3. `npm ci`
4. `npx playwright install --with-deps`
5. `npx playwright test`
6. Upload HTML report as a build artifact (retained for 30 days)

No changes to the workflow file are needed when you add new tests — it runs everything in `tests/`.

## Troubleshooting

**MCP servers not connecting**
Make sure VS Code is open at the project root (not a parent folder) so `.vscode/mcp.json` is picked up. Reload the window after any changes to that file.

**`npx playwright run-test-mcp-server` not found**
Run `npm ci` first. Playwright must be installed locally for the MCP server command to work.

**Tests fail on CI but pass locally**
Check for hard-coded `localhost` URLs or machine-specific paths. The CI environment uses `ubuntu-latest` with 1 worker and 2 retries — use the `process.env.CI` flag in your config if you need environment-specific behaviour.

**Agent does not find the user story file**
Paths in prompts are relative to the project root. Make sure your user story is saved inside `.vscode/user-stories/` and the filename in the prompt matches exactly.
